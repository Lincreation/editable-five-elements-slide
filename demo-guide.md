# 30分画面共有｜実演ガイド

## 目的

機能を網羅的に操作するのではなく、「入口を選ぶ→条件を渡す→途中を確認→直す→出力」の共通動作を見せます。

## 0〜3分｜ホームとSuper Agent

1. ホーム画面またはNewメニューを開く。
2. Super Agentと専門機能の入口を示す。
3. 「資料が欲しい」ならAI Slides、「進め方から相談」ならSuper Agentと説明する。

見るポイント：完成物が決まっているかで入口が変わる。

## 3〜15分｜AI Sheets

使用ファイル：`demo-data/Genspark研修_飲食店週次支出照合デモ.xlsx`

実演プロンプト：

> 部門別支出シートの表示順と判断基準に従い、請求書明細を分類・並べ替えてください。部門コード、部門名、判定理由、確信度、要確認を追加してください。判断材料が不足する行は推測で確定せず要確認とし、確認質問を作ってください。元の請求書IDと金額は変更しないでください。重複候補を検出し、処理前後の総額を照合してください。

確認すること：

- 部門順が基準シートと一致する。
- 判定理由が読める。
- `INV-3112`を詳細不足の要確認として残し、確認質問がある。
- `INV-3104-再送`を重複候補として残している。
- 元の請求書IDと金額が変わっていない。
- 13件、入力858,640円、出力858,640円、差額0円で一致する。

AI Sheetsを開始して5分で結果が出なければ、`demo-data/AI-Sheets_分類結果_バックアップ.xlsx`の`投影用サマリー`へ切り替えます。結果確認とXLSX出力の説明に最低5分を残します。

## 15〜23分｜AI SlidesとAI Docs

### 15〜20分｜AI Slides

AI Sheetsで確認した次のサマリーをコピーし、AI Slidesの新しい指示へ貼ります。機能間の直接連携は前提にしません。当日の契約画面に直接引き継ぐ導線がある場合だけ、補足として見せます。

> 13件 / 総額858,640円 / 例外2件。詳細不足はINV-3112、重複候補はINV-3104-再送。数値は元データと照合済み。

ライブ生成が未完なら`demo-data/AI-Sheets_分類結果_バックアップ.xlsx`の投影用サマリーから同じ文面をコピーします。

実演プロンプト：

> 以下の確認済みサマリーをもとに、店舗責任者向けの週次支出報告を3ページで作成してください。13件 / 総額858,640円 / 例外2件。詳細不足はINV-3112、重複候補はINV-3104-再送です。1ページ目は表紙、2ページ目は数値と要点、3ページ目は2件の確認事項にしてください。結論先行で、数値を変更せず、不明点は推測で補わないでください。

1. 当日のモード名と構成案を確認する。
2. 3ページ構成を承認してから生成する。
3. 2ページ目の要点だけを指定して修正する。
4. 出力場所を開き、利用可能形式は当日の契約画面に合わせて説明する。

AI Slidesを開始して2分30秒で初稿が出なければ、`demo-data/AI-Slides_週次支出報告_バックアップ.html`へ切り替えます。

### 20〜23分｜AI Docs

1. Rich Text / Markdownの入口を示す。
2. AI Slidesで使った確認済みサマリーを、新しいAI Docsの指示へ貼る。
3. 「13件・858,640円・例外2件」を短い週次共有文へまとめる。
4. `INV-3112`の確認依頼文だけを選択してAI Editする。
5. Save Pointと出力場所を示す。

AI Docsは完成まで待たず、入口・選択編集・版管理の3点をShowできれば完了です。

## 23〜26分｜画像生成・編集

1. 用途を「週次支出報告の表紙に使う16:9画像」と指定する。
2. 主役、構図、色、避けたい文字・ロゴを伝える。
3. 生成結果から一部分だけを修正する入口を示す。
4. 人物、文字、ロゴ、著作権、商品表現を人が確認すると説明する。

## 26〜30分｜SkillsとWorkflows

### Skills

- Community、My Own、Teamの表示を確認する。
- Add & UseまたはSkillBarの入口を示す。
- Skillは品質を保証するものではなく、構成・考え方・スタイルのばらつきを減らす仕組みと説明する。

### Workflows

- Trigger、Action、Test Run、Run History、Pending Confirmationを示す。
- スライドの「開始・読込・処理・分岐・確認・記録」は研修用の設計チェックで、固定ノード名ではないと伝える。
- Test Runは模擬データ、Turn On後は実アクションになり得ると説明する。
- 本研修ではTurn Onしない。
